08 - Custom Databinding

Part A. @Input and Binding Properties to Child Components

The ability to amke your properties bindable from outside, from the parent component, using the child component



  1. In child-component TS - Input imported and used as decorator to define object childComponentElement, this makes the childComponent element available outside of its singular component

  2. Child component html provides what will be displayed when the component is called within the parent, which also references the child object definition

  3. Within the parent component(custom-databinding), the element parentElements is defined, which is an object has been configured in the form to pass through the childComponentElement defined within the child. No input or special decorator required

  4. The parent component html(custom-databinding) includes the following ngFor loop binding the child defined object element(childComponentElement) to the iteration of the array of objects hosted in the parent(custom-databinding)


Part B. Using an Alias

If you want to create an alias of an object for more precise reference outside of the component an alias can be used. 'honey' is the alias used in the example


  1. In child-component TS - the 'honey' alias is defined in the () of the Input decorator

  2. In parent custom-databinding html - the 'honey' alias is referened instead of the original name of childComponentElement as the bound parameter.

Part C. @Output and Parent Component listening to Child Component then reacting

Allows parent components,using a child component, to listen to events using the event emitter.

We have shown pass data from a child component to a parent component which was executed there in part A, now in part C we go in the other direction, emmiting a listenable event from the child TS to the parent TS where then the parent component TS will run code.

Moreover once an action has occured, another component will react based on the change. So if I click a button and a popup occurs, the component will react to the change in state that a popup occured, not from the actual click of the button.

  1. Starting from the child component 'cockpit' - In cockpit, the html component includes two buttons, one that on click triggers the function onAddServer, the other triggers onAddBlueprint. Must import EventEmitter and Output at top.

    Create an @Output with an EventEmitter, defining the type of object being emmitted. The @Output and EventEmitter are similar to a listener, because they are listening to the onAdderServer/onAddBlueprint for when they get clicked, so that the EventEmitters transmit the data outside of the component.



  2. In the parent component whose html includes reference to the child component - We are EVENT BINDING to the emmitted events from the child component, serverCreated and blueprintCreated, in parenthesis seen just the same as the basic (click) event binding.

    The event bounded serverCreated and blueprintCreated listen for the event, and upon action, their own function onServerAdded and onBlueprintAdded are executed with the $event data passing into their own TS function(onServerAdded and onBlueprintAdded)



  3. The event data passed from child in the form of 'serverCreated'/'blueprintCreated' to the parent function 'onServerAdded'/'onBlueprintAdded' is then processed through the parent app component TS.
